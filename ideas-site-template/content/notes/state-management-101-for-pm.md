## {brief}

1. **状态管理的本质**：应用的数据只存一份（Store），任何改动必须发"申请单"（Action），由固定规则（Reducer）算出新结果——类比公司统一台账，禁止各部门各自记账。
2. **单向数据流是核心约束**：数据只能走"UI → Action → Reducer → Store → UI"这一条环形路径，不能抄近路。这条约束换来了可追溯、可调试、可做撤销/重做的能力。
3. **状态树按业务域拆开**：角色数据、World 数据、UI 状态各自有各自的"管家"（Reducer），互不越界改别人的分支，但合在一起仍是一棵完整的树。
4. **订阅-推送而非轮询**：Store 变了主动通知所有订阅组件，组件自检"我关心的那块数据真变了吗"才刷新，避免全局重渲染。
5. **Optimistic 乐观更新的代价**：先假设操作成功立即显示结果，背后异步确认。必须提前设计好：临时标识、视觉区分、失败回滚提示、猜错时的覆盖策略。
6. **多面板场景的核心认知**：不同性质的数据配不同的记账方式——单值用覆盖式、协作冲突用版本号、历史记录用时间序列日志；跨面板联动靠统一的广播层而非统一的仓库。
7. **对产品设计的直接启发**：写 PRD 时把"动作"和"状态变化规则"拆开写；复杂流程先画状态图；发现数据不一致时追问"这两个地方是否各自维护了数据副本"。

## {full}

### 一、为什么产品经理也该懂点状态管理

"状态管理"是前端工程里的统称——应用当前的数据和界面情况，应该怎么存、怎么改、怎么让各个页面都知道。Redux、MobX、Vuex/Pinia 是这个问题不同解法，各有取舍，背后有一套共通的思维方式。

常见痛点：
- 一个按钮点下去，页面多处同时变，牵一发动全身
- 同一数据在不同页面显示不一致
- 页面状态"诡异"，排查不出是哪一步改的
- 撤销/回退功能工程师说"现在的写法很难做"

根源往往不是 bug，而是"状态"被到处随意修改，没有统一规则。

### 二、状态管理到底在解决什么问题

**类比：没有统一台账的公司**——销售、财务、仓库各自记账，三份表经常不一致，出了事也查不出谁改的。好的状态管理方案相当于：全公司只留一份台账，任何人要改只能提交"申请单"（写清楚谁、做什么、改多少），台账管理员按固定规则算出新数字后覆盖发布。

三个具体问题：
- **数据不一致** → 唯一数据源（Single Source of Truth）
- **改动不可追溯** → 所有变化通过命名的 Action 发出
- **行为不可预测** → 规则函数相同输入永远相同输出

Redux 是这套模式最"教条"的实现（不可变数据、纯函数）；MobX 允许直接改数据、靠自动追踪依赖同步；Vuex/Pinia 按业务模块分治，模块间靠事件协调。三者回答同一个问题：状态放哪、怎么改、怎么让关心它的地方知道。

### 三、三个核心角色

**Store（仓库）**：整个应用只有一份数据源，任何页面都从这儿读，不允许各自留副本。

**Action（申请单）**：描述"发生了什么"的对象，而不是直接说结果。如 `{ type: '用户点击了发布' }`。天然适合操作留痕和行为分析。

**规则函数（Reducer）**：接收"当前状态 + Action"返回"新状态"。关键约束是纯函数——相同输入永远相同输出，不能有副作用。

### 四、单向数据流

循环：UI（用户操作）→ Action（描述发生了什么）→ 规则函数（算出新状态）→ Store（更新台账）→ UI（重新渲染）。

关键规则：**不能抄近路**。只要每次变化必经同一条路径，就可以在这条路径上做撤销/重做、操作日志、回放复现 bug 等——Redux DevTools 正是靠这条约束才做得出来。

### 五、深入一层

**5.1 状态树**：按业务域分而治之。每支有自己的"管家"，只能改自己那支。像公司分销售分账、仓库分账、财务分账，对外仍是合并总账。

**5.2 订阅-推送**：组件订阅 Store，Store 变了主动推送，组件自检"我关心的数据真变了吗"，没变不渲染。

**5.3 Optimistic 乐观更新**：先假设成功、立即展示结果，背后异步确认。四个必须提前设计的事：临时标识、视觉区分、失败回滚提示、猜错时的覆盖策略。高风险操作（支付、关键修改）谨慎使用。

### 六、真实场景代入

用户给角色「岚书」补充背景故事，走完整流程：UI 操作 → Action `{ type: '角色/更新背景故事' }` → Reducer 计算新状态 → Store 更新 → 所有引用该角色的地方（关系图、大纲、导出预览）同步刷新。

### 七、对产品设计的启发

- **动作和状态分开写 PRD**：不写"点击后变成已发布"，而是写"动作：点击发布；规则：草稿→已发布，已发布则禁用；结果：所有引用处同步"
- **先画状态图**：列出所有状态、每个状态下允许的动作、跳转关系、禁止组合
- **追问数据不一致的根因**："这两个地方是不是各自维护了数据副本？"
- **提撤销/历史记录需求前了解架构**：底层有 Action 追溯则成本低，没有则成本高很多

### 八、常见误解澄清

- Store ≠ 数据库，只是当前会话的临时状态
- 用了模式 ≠ 不出 bug，规则函数写错一样会出错
- 不是所有前端项目都需要全套状态管理，简单页面是过度设计
- Redux/MobX/Vuex 取舍不同：Redux 最严可回放最强，MobX 最顺手缺强制约束，Vuex/Pinia 介乎两者

### 九、多面板拓展场景

画布、地图、手机、邮箱四种不同性质的数据不能全塞进同一棵状态树：

- **地图标记**：单一当前值 → 覆盖式更新
- **画布节点**（多人协作）：需要版本号冲突检测
- **手机消息/邮箱**：时间序列日志，永远只新增不覆盖

**统一管理的不是存储方式，而是变化通知方式**。每个面板改完自己的账本后往广播频道发消息，其他面板各自决定要不要响应。跨面板业务规则放在独立的中间层。

同时区分"世界共享事实"（要广播要存档）和"面板临时界面状态"（不广播不存档）。

### 十、一页纸总结

- 唯一台账：一份数据源，禁止各自留副本
- 先申请后记账：发 Action 描述发生了什么，不直接改结果
- 规则透明可预测：相同输入相同输出
- 不能抄近路：单向数据流换来了可追溯可调试
- 大状态分支管、改了主动通知
- Optimistic 可以抄近路，但失败必须清楚回滚
- 不同数据配不同记账方式，同一层广播通知
- 工具是取舍不是信仰

一、为什么产品经理也该懂点状态管理
"状态管理"是前端工程里的一个统称，说的是"一个应用当前的数据和界面情况（状态），应该怎么存、怎么改、怎么让各个页面都知道"。Redux、MobX、Vuex/Pinia 都是这个问题的具体解法，各有取舍，但它们背后有一套共通的思维方式。这套思维方式，产品经理不需要会写代码也能学会，而且能直接用在做产品设计上。
下面这些场景你一定遇到过，而这些恰好是"状态管理"思路能直接帮上忙的地方：

  • "这个按钮点了之后，页面这几个地方为什么都跟着变了？改起来牵一发动全身"

  • "同一个数据，A 页面显示的和 B 页面显示的不一致，到底哪个是对的？"

  • "用户说页面状态'诡异'，工程师排查了半天说不出是哪一步改的"

  • "这个功能要支持撤销/回退，工程师说'现在的写法很难做'"

这些问题的根源，往往不是某个 bug，而是"状态"（应用当前的数据和显示情况）被到处随意修改，没有统一的规则。状态管理就是一套治理"状态该怎么改"的思路，理解它，能帮你在做需求设计、写 PRD、和工程师对齐方案时，问出更准确的问题。

──────────────────────────────────────────────────
二、状态管理到底在解决什么问题
先抛开代码，用一个类比理解"状态管理"是什么问题。
🏢 类比：一家没有统一台账的公司假设一家公司里，销售、财务、仓库各自记自己的一份"库存数量"，谁想改就直接改自己那份表。结果三份表经常不一致，出了问题也查不出是谁、什么时候改错的。好的状态管理方案的做法相当于：全公司只保留一份台账（Store），任何人要改库存，不能直接涂改台账，只能提交一张"申请单"（Action），写清楚"谁、要做什么、改动多少"。台账管理员（规则函数）收到申请单后，按固定规则算出新的台账数字，然后覆盖发布。所有部门看到的，永远是同一份、有记录可查的台账。
放回到软件产品里，状态管理想解决三个具体问题：
问题没有统一状态管理时状态管理的解法数据不一致页面上多个组件各自维护一份数据副本，容易脱节整个应用只有一份"唯一数据源"（Single Source of Truth）改动不可追溯谁在哪一行代码改了状态、为什么改，很难查所有变化都必须通过带名字的 Action 发出，每一次变化都有"名字"和"记录"行为不可预测同样的操作，因为改动逻辑分散在各处，结果可能不一致状态变化只能通过固定的"规则函数"计算，同样的输入永远产生同样的输出
需要说清楚一点：这套"唯一台账 + 申请单 + 固定规则"的思路，是一个通用模式，不是某一个工具专属的。Redux是把这个模式做得最"教条"的一种实现（严格要求不可变数据、纯函数规则）；MobX走了另一条路，允许直接修改数据，靠"自动追踪依赖"来保证界面同步更新；Vuex / 它的继任者 Pinia则是按业务模块分开管理，模块内部有统一的改法，模块之间靠事件协调。三者的具体做法不同，但都是在回答同一个问题：状态该放在哪、该怎么改、怎么让关心它的地方都知道。接下来几节讲的 Store / Action / 规则函数这套词汇，最早是 Redux 让它们变得广为人知的，但概念本身比 Redux 这个工具更通用，后面讲到具体工具选型的地方会再展开。

──────────────────────────────────────────────────
三、三个核心角色：Store / Action / 规则函数
这套模式里只有三个核心概念，理解这三个就等于理解了状态管理的全部骨架。
1️⃣ Store（仓库）— 唯一的数据源
Store是整个应用当前所有状态的"唯一台账"。任何页面要显示数据，都从这一份台账里读，不允许自己另存一份。
产品视角理解：Store 就是"当前系统的真实状态"，好比一份实时更新、唯一可信的产品数据快照。
2️⃣ Action（申请单）— 描述"发生了什么"
Action是一个简单的、带名字的对象，用来描述"发生了一件什么事"，而不是直接说"把状态改成什么"。比如、。
产品视角理解：Action 相当于一条"事件日志"，记录了用户/系统做了什么动作，而不是记录最终结果。这也是为什么这套模式天然适合做"操作留痕""行为分析"。
3️⃣ 规则函数（记账规则）— 根据申请单算出新台账
规则函数（在 Redux 里叫 Reducer）是一个函数，接收"当前状态 + 一个 Action"，返回"新的状态"。它有一个关键约束：纯函数—— 相同的输入永远得到相同的输出，且不能有副作用（不能偷偷发请求、改别的东西）。
产品视角理解：规则函数就是"业务规则"本身，比如"库存 = 原库存 - 本次出库数量"。规则写清楚了，任何人在任何时候按同样的输入算出来的结果都一样，这就是"可预测"。

──────────────────────────────────────────────────
四、单向数据流：这才是这套思路真正的核心
把三个角色串起来，就是这套模式的核心循环，也叫单向数据流：
UI 界面用户点击/操作
→
Action发出"发生了什么"
→
规则函数算出新状态
→
Store更新唯一台账
→
UI 界面重新渲染显示

关键规则只有一条：数据只能沿着这一条环形路径走，不能抄近路。UI 不能直接偷偷改 Store，必须走"发 Action → 规则函数计算 → Store 更新"这条完整流程。
💡为什么这条"不能抄近路"的规则很值钱：只要状态的每一次变化都必须经过同一条路径，你就可以在这条路径上做很多事而不用改业务代码本身，比如：记录每一次变化用于"撤销/重做"、把所有 Action 打印出来用于调试排障、甚至把某一段 Action 历史"倒带回放"复现一个 bug。Redux 生态里出名的调试工具就是靠这条约束才做得出来——正是因为路径唯一，才能像看录像一样一步步回看状态是怎么变化的。

──────────────────────────────────────────────────
五、深入一层：大状态怎么拆，怎么通知前端刷新，想抄近路时 Optimistic 怎么设计
前面四节讲的是基本模型。但实际项目一大，三个现实问题就会冒出来：状态多了怎么拆开管而不乱？数据变了页面怎么知道要刷新？如果这套流程慢得让用户觉得卡，又该怎么办？依次回答。
5.1 大状态怎么拆：状态树，按业务域分而治之
Store 里的数据不是一大盆乱炖，而是一棵树，按业务域分枝，这在状态管理里叫状态树（State Tree）：
一个 Neta Studio 可能的状态树结构
每一支都有自己的"管家"（就是前面讲过的规则函数），只能改自己那支的数据，不能伸手去改别支的。最后把所有分支的管家合在一起，拼成一棵完整的树。
🏢 类比：就像公司台账分成销售分账、仓库分账、财务分账，每本分账有自己的记账人，销售不能直接改仓库分账。但对外看，公司还是只有一本合并后的总账。
这样拆的好处是：不同开发者同时开发不同分支的逻辑，不会相互踩脚；某个分支写错了，也不会把别的分支带坏，因为权限本来就被隔开了。
5.2 怎么通知前端要刷新自己："订阅-推送"机制
状态改了，页面上十几个组件怎么知道自己该重新渲染了？答案不是"定时去问一下 Store"（轮询），而是订阅-发布（Subscribe / Publish）。

  • 每个组件在刚出现的时候，向 Store 登记"我订阅了你，你变了告诉我一声"

  • Store 一旦有新状态，会主动推送一个"我变了"的信号给所有订阅它的组件

  • 收到信号的组件会先自检："我关心的那小块数据，这次真的变了吗？"——没变就不重新渲染，变了才刷新。这一步叫"对比"（前后两份数据快照逐字比一下），目的是不让一个人的操作把整个页面都刷一遍，很耗性能

📰 类比：台账更新后，"订阅了这本台账"的部门会收到一封通知，但不是所有人都重新做一份报表——只有"自己那行数字真的变了"的部门才会重新制作自己的报表，其他部门看一眼发现无关就不动。
需要提一个产品视角的区别：这套"订阅-推送"是发生在一个应用内部（内存里）的，不跨网络。如果是"后端数据变了，前端要知道"这类跨端场景，前端自己的状态管理方案是解决不了的，常见的方案有两种：前端定时回去问一下（轮询），或者后端主动推一条消息给前端（WebSocket / 推送）。前端收到信号后，再重新抓一次最新数据、重新走一遍前面说的这套流程，本质上是同一套思路在跨端场景下的延伸。
5.3 这条路太慢了，想抄近路，Optimistic 模式怎么设计
完整走一遍"Action 发出 → 后端确认 → 状态真正更新 → 界面刷新"会有等待延迟，用户点一下发布，很可能的体验是"卡一下才反应"。Optimistic（乐观更新）就是为了解决这个"卡感"而抄的那条近路：先假设这次肯定会成功，立即在界面上把结果改好给用户看，同时悬后台悄悄地跟后端确认。如果后端说不行，再把界面"倒带"回去并提醒用户。
类比：餐厅下单你在餐厅小程序上点"确认下单"，页面立即显示"已接单，制作中"，但这个时刻后厨可能还没收到。页面先"乐观地假设会接单成功"展示了结果，背后静静发个请求给后厨确认。请求真正失败了（比如餐厅已打烊），才会把页面改回"下单失败，请重试"。

设计 Optimistic 模式时，需要提前想清楚四个问题，这四个也是它"代价"的所在：
需要提前设计的事为什么必要给每个"乐观操作"标个暂时的作记号后端的真实结果回来时，需要知道"这个结果对应的是刚刚那个乐观操作"，才能把暂时结果换成真实结果，而不是拿错地方界面上要能看出"这个是暂时的"比如变淡、加个小 loading 图标，不要让用户以为"已完全确定"，不然一旦回滚会很困惑"刚刚那个去哪了？"想好失败了怎么回滚失败时不能静静把内容拿掉了事，必须同时告诉用户"刚刚那个失败了，因为 XX"，否则用户会觉得自己的内容"诡异地"消失了想好"猜错了怎么办"如果前端乐观猜的结果和后端真实结果不一样（比如库存被别人先抢了），要提前定义"谁说了算"——通常是后端的真实结果最终覆盖前端的乐观猜测
⚠️产品视角的提醒：Optimistic 本质上是用"技术复杂度"换"用户感知延迟"，不是默认适合所有场景。提这类需求时，先问自己：这个操作失败的代价大吗？失败概率高不高？像"点赞""关注"这类几乎不会失败、失败了也不严重的操作，很适合乐观更新；而"支付""提交关键修改"这类失败代价大、频率不低的操作，要谨慎使用，或者至少把回滚提醒设计得够明显。

六、用 Neta Studio 的真实场景走一遍
用一个大家都熟悉的场景代入：用户在 World 编辑器里给一个角色补充设定。
场景：用户给角色「岚书」补充一段背景故事
这个例子能帮你直观看到几件事：

  • 为什么"保存角色设定"之后，World 里其他引用这个角色的地方（关系图、剧情大纲）都能同步更新——因为它们读的是同一份 Store，不是各自复制的数据

  • 为什么产品提"支持撤销上一步编辑"这类需求时，工程师如果是走这套思路做的，成本会低很多——因为每一次变化都是一条可追溯、可回放的 Action

  • 为什么"同一个操作，结果却不一样"是个危险信号——如果规则函数是纯函数，这种情况本不该发生，出现了往往说明代码里有偷偷绕开 Store 的"抄近路"逻辑

七、这套思路对产品设计的启发
即使完全不碰代码，这套思维方式也可以直接用在做产品设计上。
7.1 写需求时，把"状态"和"动作"分开描述
很多 PRD 把"用户做了什么"和"页面变成什么样"混在一句话里，容易漏边界情况。试着拆开写：
不推荐的写法拆开后的写法"用户点击发布后页面变成已发布状态"动作：用户点击"发布"按钮规则：若当前状态为"草稿"，则变为"已发布"；若已是"已发布"，点击应禁用/无效果结果：所有引用该状态的地方（列表页标签、详情页按钮、通知）同步更新
7.2 设计"状态图"，把所有状态和触发条件列全
这套思路逼着工程师把"状态"和"能触发状态变化的动作"枚举清楚。产品经理写复杂流程（审核、发布、协作编辑）时，同样应该先画一张状态图，列出：

  • 有哪些状态（草稿 / 待审核 / 已发布 / 已下线……）

  • 每个状态下，允许发生哪些动作

  • 每个动作发生后，会跳到哪个状态

  • 有没有"不允许发生"的组合，需要在交互上禁用或提示

这类图工程师一般会很喜欢，因为它跟"动作 → 规则函数 → 新状态"的思路完全对应，能直接翻译成实现方案，减少来回澄清的成本。
7.3 认识到"唯一数据源"对产品体验的价值
当你在评审时发现"这个数字在两个页面显示不一致"，别只当成一个 bug 提给工程师，可以多问一句：这两个地方是不是各自维护了一份数据副本？如果是，说明系统里缺一个"唯一数据源"，这类问题会反复出现，值得作为技术债推动统一，而不是每次单独修一个显示。
7.4 提"撤销""历史记录""协作冲突处理"类需求时，先了解现状架构
撤销/重做、操作历史、多人协作冲突检测，这些需求如果底层是这套思路做的状态管理，实现成本通常较低（因为每一步变化本来就是可追溯的 Action）。如果底层不是这么做的，同样的需求实现成本可能高很多。提需求前跟工程师确认一下"我们现在状态是怎么管理的"，能帮你更合理地评估排期，而不是"这个需求听起来很简单，怎么要这么久"。

八、常见误解
误解实际情况Store 就是"存数据的地方"，跟数据库差不多Store 只是"当前这一次会话里、这个应用界面正在展示和使用的临时状态"，不是持久化存储。用户刷新页面，Store 通常会重置，真正需要保存的数据还是要落到后端数据库。用了这套模式就自动"不会出 bug"这套模式只是约束了"改状态的方式"，如果规则函数写错了逻辑，一样会出错，只是排查起来会比"到处随便改状态"容易得多。所有前端项目都必须上这一整套状态管理方案这套思路更适合状态复杂、多个页面/组件共享同一批数据的中大型应用。简单页面上一整套状态管理反而是过度设计，工程师选型时会综合考虑复杂度，不是"越正规越好"。Redux、MobX、Vuex 是同一个东西，随便用哪个都一样它们解决的问题一样，但取舍不同：Redux 要求数据不可变、靠对比新旧引用判断要不要刷新，规矩最严、可回放能力最强；MobX 允许直接改数据、靠自动追踪依赖刷新界面，写起来最顺手，但缺少"必须走统一入口改状态"的强制约束；Vuex/Pinia 按业务模块划分，模块内部有统一的改法，介于两者之间。选哪个要看团队已有的技术栈和项目复杂度，不是无脑选"最出名的那个"。

九、拓展场景：画布、地图、手机、邮箱同时开在一个 World 里，怎么设计
前面几节讲的是"用一套状态管理方案统一管所有状态"这个基本模型。但如果照搬这个模型去设计一个真实的多面板 World，会掉进一个新的坑：把画布、地图、手机、邮箱的数据全都塞进同一棵状态树，用同一套规则去改。这一节要纠正这个想法——统一不是"塞进一个仓库"，而是"每类数据挑一种跟它本质匹配的记账方式，账本之间靠一层统一的通知机制打通，而不是逼它们共用同一本账"。
9.1 场景：四个面板，性质完全不一样
想象用户在这个 World 里做了四件事：

  • 在画布上，跟另一个人同时拖动同一个角色的位置（多人协作，随时可能冲突）

  • 在地图上给一个地点做了标记，取名"废弃教堂"（相对简单，改了就是改了）

  • 打开角色的手机，给另一个角色发了条消息（这是一条不断累加的时间线，不是"当前值"）

  • 过一会儿去看邮箱，发现收到一封新邮件（同样是往时间线里加一条，不是覆盖）

这四类数据表面上都是"World 里的数据"，但骨子里完全不同性质，如果不分青红皂白都塞进同一棵状态树、用同一套"整体覆盖式"的规则改，会出问题：
一刀切会踩的坑
  • 画布的坑：两个人同时拖同一个角色，如果规则是"谁的改动后到就整体覆盖"，会丢掉其中一个人的操作，且没人知道丢了什么、什么时候丢的

  • 手机/邮箱的坑：聊天记录、邮件本质是"一条条往后加的历史"，如果套用"整体覆盖"的规则去存，等于每次都要把全部历史记录重新整个搬一遍，即使只是新增一条消息

  • 地图的坑反而不大：地图标记这种数据确实比较适合"整体状态、改了就覆盖"的简单模型

9.2 关键认知：不是"一个大仓库"，而是"每类数据配一种匹配的记账方式"
回到第二节"公司统一台账"的类比，但要把它讲得更准确：一家大公司不会把销售、仓库、工程图纸、邮件往来全塞进同一张 Excel 表用同一套记账规则。真实情况是——销售用简单的进出账表（改了就覆盖最新值就行）；工程图纸要用"版本历史+锁"的方式管理（防止两个工程师同时改同一张图互相覆盖）；邮件往来天然是一条条按时间顺序累加的记录，永远是"新增一条"而不是"覆盖旧的"。每种数据用它天生适合的记账方式，而不是逼着所有部门用同一张表格。
放回 World 的四个面板，每类数据的"天生性质"决定了它该用哪种记账方式：
面板数据的天生性质该用的记账方式类比地图标记、角色所在位置相对简单，某一时刻只有一个"当前值"整体状态 + 覆盖式更新（就是前面几节讲的标准模型）普通台账，改一下数字就行画布上的节点（多人同时拖动、编辑）会有多人同时改同一份内容，需要知道"谁在旧版本基础上改的"，避免互相覆盖带版本号的操作记录（每次改动带上"基于哪个版本改的"，冲突了能检测出来，不是谁后到谁说了算）工程图纸的版本管理，每次改动要登记"基于第几版改的"手机消息、邮箱邮件天生是按时间顺序累加的历史记录，从不覆盖，只有"新增"时间序列日志（每条消息/邮件是一条独立记录，永远往后加，不改旧记录）银行流水，从不覆盖旧的一条，只会新增一条
这里最容易犯的错误，就是看到"要统一管理多个面板的状态"，就直接联想到"那就用一个大 Store 存所有东西"。统一管理的意思不是"存储方式也要统一"，而是"变化的通知方式要统一"——这是下一节要讲的。
9.3 各自记账，但改动要能互相通知：统一的广播层，不是统一的仓库
四类数据用不同的记账方式存好之后，还有个问题：画布上角色挪了位置，地图上这个角色的坐标点也该跟着动；手机发了消息，也许要触发邮箱收到一封新邮件。这种"跨面板联动"要怎么做，答案不是"让地图直接去读画布的数据结构"（两者的记账方式完全不同，地图根本读不懂画布的版本号规则），而是：每个面板改完自己的账本之后，往一个共同的广播频道里说一声"我这边发生了什么"，其他面板各自决定要不要理会这条广播。
广播层怎么运作
关键点：广播层本身不存业务数据，它只是一条"发生了什么"的消息通道。画布的版本历史、地图的当前坐标、手机的消息流、邮箱的邮件列表，各自仍然在自己最适合的记账本里，广播层不会、也不该把这些不同性质的数据拍平成一种格式硬塞进同一个仓库。这跟前面刚说的"每种数据配一种记账方式"不矛盾——"发生了什么"的广播格式可以统一，但"这个事实最终记在哪本账、用什么规则记"，仍然按各自数据的性质走。
📻 类比：还是电台广播和收音机。画布"广播"了一句"角色挪到了 (120, 80)"，但画布自己那本"版本历史账"、地图自己那本"当前坐标账"，是两种完全不同的记账格式，谁都不需要理解对方账本的记法。广播出去的只是一句人话，各收音机（面板）自己决定听了之后要不要在自己的账本上记一笔，用什么方式记。
9.4 更复杂的联动：一个广播可能牵出另一个广播
再看"手机发消息可能触发邮箱收到新邮件"这个例子。这里不适合让邮箱直接监听"手机发消息"这条广播就直接生成邮件——因为中间往往有业务规则（比如"只有对方当前不在线才转成邮件"）。更合适的做法是：在广播层和各面板之间，加一段专门处理"跨面板规则"的中间逻辑，它监听第一条广播，判断完规则后，自己决定要不要再发出第二条广播。
广播链：发消息 → 判断规则 → 生成新邮件
这样拆的好处是：手机和邮箱两个面板互相都不需要认识对方，都只需要认识广播层；跨面板的业务规则单独放在中间这一层，以后规则变了（比如"改成不管在不在线都发邮件提醒"），只需要改这一层，不用动手机或邮箱任何一个面板的代码。
9.5 别把面板自己的"临时状态"也塞进共享账本
还有一类容易踩的坑：地图当前放大到几倍、邮箱当前打开的是哪个文件夹，这些是面板自己的界面状态，跟其他面板完全无关，也不需要广播出去、更不需要存进这个 World 的存档里。但角色现在在哪、发生过哪些邮件，才是整个 World 真正的共享事实，需要广播给关心它的面板，也需要存进存档、下次打开还在。
类型举例要不要广播要不要存档世界共享事实角色位置、地图标记、邮件内容、聊天记录要，别的面板可能关心要面板自己的临时界面状态地图缩放层级、邮箱当前打开的文件夹、画布当前选中的工具不要，跟其他面板无关不要
💡这一层对产品设计的启发：设计"多个面板/多个 work 同屏协作"类需求时，别急着问"要不要做成一个统一的大 Store"，先按数据本身的性质问三个问题：①这份数据某一时刻只有一个值，还是会有多人同时改、需要识别冲突，还是天生是一条条往后加的历史记录？②除了它自己的面板，还有没有别的面板关心它变了？③下次打开 World，还需要记得吗？第一个问题决定这份数据该配哪种记账方式（覆盖式 / 带版本冲突检测 / 时间序列日志），第二、三个问题决定它要不要广播、要不要存档。评审这类需求时，建议直接让工程师把每类数据按这三个问题过一遍、列一张表，比笼统地说"要统一管理状态"能对齐得准得多。

十、一页纸总结
记住这几句话就够了

  • 唯一台账：整个应用的状态只存一份，谁都从这一份读，不允许各自留副本

  • 先申请、后记账：任何变化都先发一个"说清楚发生了什么"的 Action，不能直接改结果

  • 规则透明、可预测：改状态的规则函数必须是固定的、相同输入相同输出，不能看运气

  • 不能抄近路：数据永远沿"动作 → 规则计算 → 更新台账 → 界面刷新"这一条路走，这条约束换来了可追溯、可调试、易扩展撤销/历史等能力

  • 大状态要分支管、改了要主动通知：状态树按业务域拆开，每支只由对应的规则改；数据变了主动推送给关心它的地方，不让别人定时回来问

  • 抄近路（Optimistic）可以，但要想好失败怎么回滚：先乐观展示结果能提升体验，但失败时必须能清楚回滚并告诉用户原因，高风险操作要谨慎使用

  • 不同性质的数据，配不同的记账方式，但用同一层广播通知：单一值用覆盖式，多人并发编辑用带版本号的记录，历史记录用时间序列日志；各自记好自己的账，但改动要往同一个广播频道发一声，让关心它的地方自己决定要不要跟着反应

  • 工具是取舍，不是信仰：Redux、MobX、Vuex/Pinia 只是这套思路的不同实现，选型要看团队技术栈和项目复杂度，不是哪个"更正规"就选哪个

📌给产品同学的下一步建议：下次评审复杂交互流程或提"状态相关"的需求（审核流、协作编辑、撤销历史）时，试着先画一张"状态 + 动作"的流转图再进评审，能显著减少和工程师来回澄清的轮次。
