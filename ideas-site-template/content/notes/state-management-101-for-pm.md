## {brief}

1. **状态管理的本质**：应用的数据只存一份（Store），任何改动必须发"申请单"（Action），由固定规则（Reducer）算出新结果——类比公司统一台账，禁止各部门各自记账。
2. **单向数据流是核心约束**：数据只能走"UI → Action → Reducer → Store → UI"这一条环形路径，不能抄近路。这条约束换来了可追溯、可调试、可做撤销/重做的能力。
3. **状态树按业务域拆开**：角色数据、World 数据、UI 状态各自有各自的"管家"（Reducer），互不越界改别人的分支，但合在一起仍是一棵完整的树。
4. **订阅-推送而非轮询**：Store 变了主动通知所有订阅组件，组件自检"我关心的那块数据真变了吗"才刷新，避免全局重渲染。
5. **Optimistic 乐观更新的代价**：先假设操作成功立即显示结果，背后异步确认。必须提前设计好：临时标识、视觉区分、失败回滚提示、猜错时的覆盖策略。
6. **多面板场景的核心认知**：不同性质的数据配不同的记账方式——单值用覆盖式、协作冲突用版本号、历史记录用时间序列日志；跨面板联动靠统一的广播层而非统一的仓库。
7. **对产品设计的直接启发**：写 PRD 时把"动作"和"状态变化规则"拆开写；复杂流程先画状态图；发现数据不一致时追问"这两个地方是否各自维护了数据副本"。

## {full}

### 一、为什么产品经理也该懂点状态管理

"状态管理"是前端工程里的统称——应用当前的数据和界面情况，应该怎么存、怎么改、怎么让各个页面都知道。Redux、MobX、Vuex/Pinia 是这个问题不同解法，各有取舍，背后有一套共通的思维方式。

常见痛点：
- 一个按钮点下去，页面多处同时变，牵一发动全身
- 同一数据在不同页面显示不一致
- 页面状态"诡异"，排查不出是哪一步改的
- 撤销/回退功能工程师说"现在的写法很难做"

根源往往不是 bug，而是"状态"被到处随意修改，没有统一规则。

### 二、状态管理到底在解决什么问题

**类比：没有统一台账的公司**——销售、财务、仓库各自记账，三份表经常不一致，出了事也查不出谁改的。好的状态管理方案相当于：全公司只留一份台账，任何人要改只能提交"申请单"（写清楚谁、做什么、改多少），台账管理员按固定规则算出新数字后覆盖发布。

三个具体问题：
- **数据不一致** → 唯一数据源（Single Source of Truth）
- **改动不可追溯** → 所有变化通过命名的 Action 发出
- **行为不可预测** → 规则函数相同输入永远相同输出

Redux 是这套模式最"教条"的实现（不可变数据、纯函数）；MobX 允许直接改数据、靠自动追踪依赖同步；Vuex/Pinia 按业务模块分治，模块间靠事件协调。三者回答同一个问题：状态放哪、怎么改、怎么让关心它的地方知道。

### 三、三个核心角色

**Store（仓库）**：整个应用只有一份数据源，任何页面都从这儿读，不允许各自留副本。

**Action（申请单）**：描述"发生了什么"的对象，而不是直接说结果。如 `{ type: '用户点击了发布' }`。天然适合操作留痕和行为分析。

**规则函数（Reducer）**：接收"当前状态 + Action"返回"新状态"。关键约束是纯函数——相同输入永远相同输出，不能有副作用。

### 四、单向数据流

循环：UI（用户操作）→ Action（描述发生了什么）→ 规则函数（算出新状态）→ Store（更新台账）→ UI（重新渲染）。

关键规则：**不能抄近路**。只要每次变化必经同一条路径，就可以在这条路径上做撤销/重做、操作日志、回放复现 bug 等——Redux DevTools 正是靠这条约束才做得出来。

### 五、深入一层

**5.1 状态树**：按业务域分而治之。每支有自己的"管家"，只能改自己那支。像公司分销售分账、仓库分账、财务分账，对外仍是合并总账。

**5.2 订阅-推送**：组件订阅 Store，Store 变了主动推送，组件自检"我关心的数据真变了吗"，没变不渲染。

**5.3 Optimistic 乐观更新**：先假设成功、立即展示结果，背后异步确认。四个必须提前设计的事：临时标识、视觉区分、失败回滚提示、猜错时的覆盖策略。高风险操作（支付、关键修改）谨慎使用。

### 六、真实场景代入

用户给角色「岚书」补充背景故事，走完整流程：UI 操作 → Action `{ type: '角色/更新背景故事' }` → Reducer 计算新状态 → Store 更新 → 所有引用该角色的地方（关系图、大纲、导出预览）同步刷新。

### 七、对产品设计的启发

- **动作和状态分开写 PRD**：不写"点击后变成已发布"，而是写"动作：点击发布；规则：草稿→已发布，已发布则禁用；结果：所有引用处同步"
- **先画状态图**：列出所有状态、每个状态下允许的动作、跳转关系、禁止组合
- **追问数据不一致的根因**："这两个地方是不是各自维护了数据副本？"
- **提撤销/历史记录需求前了解架构**：底层有 Action 追溯则成本低，没有则成本高很多

### 八、常见误解澄清

- Store ≠ 数据库，只是当前会话的临时状态
- 用了模式 ≠ 不出 bug，规则函数写错一样会出错
- 不是所有前端项目都需要全套状态管理，简单页面是过度设计
- Redux/MobX/Vuex 取舍不同：Redux 最严可回放最强，MobX 最顺手缺强制约束，Vuex/Pinia 介乎两者

### 九、多面板拓展场景

画布、地图、手机、邮箱四种不同性质的数据不能全塞进同一棵状态树：

- **地图标记**：单一当前值 → 覆盖式更新
- **画布节点**（多人协作）：需要版本号冲突检测
- **手机消息/邮箱**：时间序列日志，永远只新增不覆盖

**统一管理的不是存储方式，而是变化通知方式**。每个面板改完自己的账本后往广播频道发消息，其他面板各自决定要不要响应。跨面板业务规则放在独立的中间层。

同时区分"世界共享事实"（要广播要存档）和"面板临时界面状态"（不广播不存档）。

### 十、一页纸总结

- 唯一台账：一份数据源，禁止各自留副本
- 先申请后记账：发 Action 描述发生了什么，不直接改结果
- 规则透明可预测：相同输入相同输出
- 不能抄近路：单向数据流换来了可追溯可调试
- 大状态分支管、改了主动通知
- Optimistic 可以抄近路，但失败必须清楚回滚
- 不同数据配不同记账方式，同一层广播通知
- 工具是取舍不是信仰
